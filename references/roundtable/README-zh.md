# Roundtable Operator（LensTable）

collab-master 的发散模块在碰到复杂问题时，有时候也会卡住———给出来的几个方案看着不同，细看是一个意思。Roundtable 就是用来解决这种情况的，普通发散不够的时候，它拉一桌人，各自用不同的眼光盯着同一个问题，再把真正的分歧挑出来，弱的淘汰，好的留下。

当前 v0.2。30 张思考镜头可用，6 个角色，两档运行。

## 和"直接让几个 AI 各自回答"有什么区别

直接让多个 AI 回答同一个问题，会产生"看起来有分歧"的幻觉，而不是真正的分歧。原因是系统性的：

| 没有结构的多 AI | Roundtable 的答法 |
|---|---|
| 模型训练来源相似 → 倾向于达成一致 → 假发散 | 每个角色有**禁区**：越界的发言当场作废，不进入裁判的综合 |
| 没有可证伪性要求 → 观点堆积，没有一条被检验 | `falsificationTest` **强制填写**：没有这条的 session 无效，不计数 |
| 角色漂移——"管落地的人"开始替用户说话 | 越域执法：该轮发言无效，裁判排除 |
| 汇总剧场——裁判只是把所有人的话归纳一遍 | 裁判必须淘汰方案、记录真实冲突、解释选择理由——缺一项，结果无效 |

结构上的根本差别：没有结构的多 AI 产的是更多的文字，Roundtable 产的是**可追溯的真实分歧**——被淘汰的是哪条、为什么淘汰、剩下的分歧是事实层面还是价值层面的、最终选了哪个方向为什么不选别的。这几样缺一样，这桌讨论在 collab-master 的质量闸里是无效的。

**它在 collab-master 里的位置**：Roundtable 不是一个平级模块，它是 solution-space 里的升级路径。常规发散先跑；只有发散判定失败（角度太少、没有反直觉选项、所有方向落在同一类）才开 Roundtable。大多数任务根本不会走到这里。

## 它解决一个什么问题

AI 在"多想几个方案"这件事上有个隐藏毛病：列五个选项，名字不一样，内核是同一个。比如让你推荐礼物，它列了"送花""送巧克力""送香水"——三个都是"买一个东西送过去"。没有一个是"不买东西，带她去她一直想去的地方"，更没有一个是"把她过去一年聊天里说过的最有意思的一百句话做成一本小书"。

Roundtable 做的事，就是逼出这种**跳出了同一个框的东西**。它让不同的人（角色）用不同的眼光（镜头）同时盯着同一个问题，然后挑一个裁判（chair）揪出谁在跟风、谁说了废话、谁提了真东西，最后决定：哪个方向值得往下走，哪个一开始就该毙掉。

上面的例子是真人跑过的——三个角色给了三个答案：user-advocate 说买花买巧克力，implementer 说送键盘，critic 说如果她最讨厌的就是键盘呢。chair 没有选花、巧克力、键盘里的任何一个。它说：三个答案背后有同一种假设——"礼物必须是买的"——先把这个假设毙掉，再从头想。

## 什么时候用它

不是所有问题都需要拉一桌人。只有在"普通发散确实没跑出新东西"的时候才开。具体来说，要同时满足三件事：

1. 问题的复杂度够高（boundary 判定为高复杂）
2. 任务的模式是创意发散、深度分析或排查问题（CREATIVE / DEEP / DEBUG）
3. 前面跑的那轮发散已经判定失败——给出来的角度不够多、没有反直觉的选项、或者所有方向落在同一类里

反过来，这几件事只要有一件成立就不开：
- 发散的结果已经够好（已经有反直觉方向了）
- 就是个查事实的问题或者简单的代码小改
- 你说了"直接给结论"
- 医疗、法律、金融这种高风险领域——不能让角色扮演替代专业边界判断

## 怎么跑（两档）

一档叫 light（轻量），一档叫 full（全面）。

light 在单个模型内部跑。模型自己扮演 3 到 4 个角色，各用各的视角——替用户说话的人、想怎么落地的人、专门挑刺的人、负责给出奇兵的人。说完一轮之后，裁判（chair）揪出谁说了一样的话、谁越了界、谁提了真东西，然后收束。不另外起 agent，省 token。

full 给每个角色开独立的 agent。每个 agent 有自己完整的上下文，不互相看到对方的推理过程。而且每人可以戴一个"思考镜头"——比如挑刺的那个人同时用"预演失败"的镜头来想问题。这比 light 贵很多，但只有这样才能产生真正的分歧，因为角色之间不会被彼此的思路带跑。

light 有个风险要诚实讲：因为所有角色共享同一个模型的大脑，容易说着说着就说到一块去了。如果裁判发现大家的结论一句话就能概括——这个 roundtable 白跑了，自动升到 full 或者直接判失败。

## 六个角色

每个角色有自己的地盘（能管什么）和禁区（踩了什么线就作废）。

**替用户说话的人（user-advocate）**。管用户需求、使用场景、痛点在哪儿、现在在用什么东西凑合、跟什么比更愿意换。不能碰："这个东西技术上能做 X""用户想要 Y"（想替用户表态得先问过用户自己）。

**想怎么落地的人（implementer）**。管能不能做、花多少钱、技术约束、架构怎么选、性能边界。不能碰："用户想要 X"——那是替用户说话的人的地盘。也不能跳过物理约束直接给理想方案。

**专门挑刺的人（critic）**。管拆穿假设、暴露盲点、预演失败、检验能不能被证伪。不能说"这不对"就跑了——得说清楚"怎样才算对"。也不能凭空编一个前面没人提过的新事实来攻击别人。

**负责出“奇兵”的人（divergent-thinker）**。管跳出惯性、反直觉的选项、从别的领域借类比、重新框定问题本身。不能拿"创造性"当借口输出已经被否决过的路径。丢一个名字不给推理链也不行——得说清楚怎么通的。

**管价值风险的人（steward）**。只有高风险任务或者涉及对"人"的自动决策时才叫上他。管伦理风险、代价摊到谁头上、长远后果、弱势一方的处境。不能拿"这样做才对"架空落地的可行性，也不能替代 boundary 对专业的判断。

**裁判（chair）**。常在。管收束、拒绝、选方向、抽分歧、保留没解的争议。不能自己编一个前面没人提过的新主意——裁判不负责上场比赛。也不能输出用户能看的文字——他产的是内部工件，compose 模块负责把工件转成你能读的话。

## 三十种思考镜头

镜头是可选的外挂。full 档里每个角色可以戴一个，用来加深他看问题的角度。

比如挑刺的人戴上"预演失败"的镜头：他不是一般地攻击，而是先假设方案已经失败了，倒推为什么会失败，再按概率排序把死因列出来。管落地的人戴上"第一性原理"的镜头：他不是给你推"行业标准做法"，而是先把每一个"大家默认这么做的"前提找出来，再逐条问：这个前提在这里真的成立吗？

这些镜头不是让角色"装得像某个思想家"——镜头不教人怎么说话，只规定他必须产出什么。每个镜头有一组硬性要求（output_obligation）：没产出这些，就等于没真用镜头，整个 session 作废。

目前有 30 张镜头卡，涵盖 10 类：认识论（怎么知道一个判断是对的）、系统思维（怎么看到反馈和延迟）、经济激励、权力制度、伦理价值、心理行为、设计产品、工程执行、美学叙事、东方思想。

## 裁判怎么干活

裁判收到所有人的发言之后，先做三件事，再做一件事。

先做的三件：

第一，查谁闯了别人的地盘。替用户说话的人跳出来说"这个技术上能做"——越界，该轮发言作废。挑刺的人说"这不对"但是没给出"怎样算对"——也是越界。

第二，查有没有人在唱同一出戏。两个人说的核心判断只是换了措辞，裁判标为重复。三个或更多人用了同一种认知框架（全是"一层一层往下钻"，没人用"循环反馈"或"光谱拉扯"），说明这桌人集体掉进了同一个思维惯性——发散失败，重来。

第三，如果所有人的核心判断能用一句话概括，这桌讨论就是表演——裁判判失败。

做完前面的清理，裁判才开始干活：谁的主张被淘汰，为什么淘汰；真正有冲突的地方是哪些——是事实分歧还是价值分歧；哪几个主张可以拼在一起用；最终决定往哪个方向走，为什么选它不选别的；还有哪些事没争明白，留给下一轮。

有几条硬规矩卡着裁判：必须淘汰掉一些主张（一个都不淘汰说明没干活），必须在不同维度上记录真实的冲突，必须说清楚自己选这个方向的理由。三样缺一样，裁判的结果无效。

## 适合用在什么地方

- 一个方案怎么选都差不多，或者怎么选都差点意思——需要有人跳出框架重新看问题
- 要评估一个决策值不值得做，有没有没看到的代价
- 需要同时考虑技术、用户、商业、伦理，而且这几个维度之间有冲突
- 起名、设计方向、品牌调性——单一个人想容易卡在最初几个想法里，需要多方拉扯
- 架构选型、迁移策略——选错了代价大，需要从多个侧面先攻击一轮再定

不适合的情形也很清楚：查社保余额、修一行拼写错误、你说了"直接给答案别磨叽"、或者前面那轮发散已经给了一个反直觉的好方向——这些时候，拉一桌人纯粹是浪费。

## 怎么验证它不是花架子

三条硬标准，任何时候都生效：

1. 裁判的结果里必须有人被淘汰。没有人被淘汰，说明所有人都给了一样的答案，等于没发散。
2. 裁判必须记录真实的分歧。没有分歧，说明角色只是在用不同措辞说同一件事。
3. 裁判必须解释自己为什么选这个方向。只给选择不给理由，等于"我觉得这个好"——不算理由。

另外每个角色的发言还得附带一条可证伪的断言——不是"如果证据相反就错了"这种说了等于没说的大白话，而是一条具体的、能做实验验证的判断。拿不出这条，发言当场无效。

已经在三个真实任务上跑过验证：给程序员冥想 App 起名字、AI 筛选简历要不要上、微服务架构要不要拆。三个案例都产出了真正的分歧、淘汰了弱的方案、选出的方向不是因为"大家一致同意"，而是"各有各的保留，但 chair 的判断可以追溯"。
